# Workflow ระบบขายประกันภัยรถยนต์ผ่าน LINE OA

เอกสารนี้อธิบายลำดับการทำงานตั้งแต่ลูกค้าเข้าระบบผ่าน LINE OA จนจบกระบวนการขาย เพื่อใช้ตรวจสอบว่าระบบทำงานถูกต้องครบถ้วนหรือไม่

## ภาพรวมสั้น

เส้นทางหลักของลูกค้า:

```text
LINE OA
-> LIFF Gateway
-> หน้าเว็บประกันรถยนต์
-> ส่งฟอร์มขอราคา
-> บันทึก Lead
-> เจ้าหน้าที่ติดตาม
-> ออกใบเสนอราคา
-> ลูกค้าเลือกแผน
-> รับเอกสาร/ชำระเงิน
-> ออกกรมธรรม์
-> บริการหลังการขาย/ต่ออายุ
```

## ผู้เกี่ยวข้องในระบบ

- ลูกค้า: คนที่เข้ามาจาก LINE OA เพื่อขอราคา ต่อประกัน หรือดูโปรโมชั่น
- LINE OA: ช่องทางหลักที่ลูกค้ากด Rich Menu, Broadcast, ข้อความต้อนรับ หรือ Flex Message
- LIFF Gateway: หน้ากลางสำหรับ login ผ่าน LINE แล้วส่งลูกค้าไปหน้าเป้าหมาย
- หน้าร้านประกันรถยนต์: `car-insurance.html`
- Google Sheets / CRM: ที่เก็บข้อมูล lead และงานติดตาม
- เจ้าหน้าที่ขาย: คนติดต่อลูกค้า ออกใบเสนอราคา และปิดการขาย
- แอดมิน: คนดูแลโปรโมชั่น แผนประกัน และข้อมูลหลังบ้าน

## Workflow ละเอียด

### 1. ลูกค้าเริ่มจาก LINE OA

ลูกค้าเห็นช่องทางใดช่องทางหนึ่งใน LINE OA:

- Rich Menu: ขอราคาประกันรถยนต์, โปรโมชั่น, ต่อประกัน, คุยกับเจ้าหน้าที่
- Broadcast: ข้อความโปรโมชัน
- ข้อความตอบกลับอัตโนมัติ
- Flex Message พร้อมปุ่มกด
- QR code ที่พาเข้า LINE OA แล้วเปิดลิงก์ LIFF

ลิงก์ที่ควรใช้:

```text
https://liff.line.me/2010419567-tIgtTpB2?url=https%3A%2F%2Fichanachock.github.io%2FLifeInsurance-System%2Fcar-insurance.html
```

สิ่งที่ต้องตรวจสอบ:

- ปุ่มใน LINE OA ต้องเป็น action แบบ `URI` หรือ `Open URL`
- ต้องใช้ลิงก์ `https://liff.line.me/...` ไม่ใช่ลิงก์ `car-insurance.html` ตรง ๆ ถ้าต้องการให้ผ่าน LINE login
- เปิดจากแอป LINE บนมือถือจริงได้
- เปิดจาก iPhone และ Android ได้

### 2. ระบบเปิด LIFF Gateway

เมื่อกดลิงก์จาก LINE OA ระบบจะเปิด LIFF App แล้วเข้าหน้า:

```text
https://ichanachock.github.io/LifeInsurance-System/liff-gateway/
```

หน้าที่ของ LIFF Gateway:

- เรียก `liff.init()` ด้วย LIFF ID
- ตรวจว่าลูกค้า login LINE แล้วหรือยัง
- ถ้ายังไม่ login ให้พาไป login
- จำ URL ปลายทางจาก parameter `url`
- หลัง login สำเร็จ ให้ redirect ไป URL ปลายทาง
- ถ้าเปิดผ่าน LINE และตั้งค่า Add friend option / Bot Prompt ไว้ ระบบสามารถชวนเพิ่มเพื่อน OA ได้

สิ่งที่ต้องตรวจสอบ:

- `LIFF ID` ใน `liff-gateway/index.html` ต้องเป็น `2010419567-tIgtTpB2`
- Endpoint URL ใน LINE Developers ต้องเป็น `https://ichanachock.github.io/LifeInsurance-System/liff-gateway/`
- Scope ต้องมี `profile` และ `openid`
- หลัง login แล้วค่า `url` ต้องไม่หาย
- ถ้าไม่มี parameter `url` ควรมี fallback หรือข้อความแจ้งผู้ใช้

### 3. ลูกค้าเข้าสู่หน้าร้านประกันรถยนต์

หลังผ่าน LIFF แล้ว ลูกค้าจะถูกส่งไปที่:

```text
https://ichanachock.github.io/LifeInsurance-System/car-insurance.html
```

หน้าร้านควรทำหน้าที่:

- แสดงข้อมูลประกันรถยนต์ให้เข้าใจง่าย
- แสดงประเภทประกัน เช่น ชั้น 1, 2+, 3+
- แสดงจุดเด่นและบริการ
- แสดงโปรโมชั่นที่ต้องการขาย
- มีปุ่มขอราคา/เช็กเบี้ยที่ชัดเจน
- มีฟอร์มสำหรับเก็บข้อมูล lead

สิ่งที่ต้องตรวจสอบ:

- หน้าโหลดได้เร็วบนมือถือ
- ปุ่มขอราคาเลื่อนไปที่ฟอร์มถูกต้อง
- ข้อความภาษาไทยอ่านถูกต้อง
- รูป hero แสดงผลได้
- ฟอร์มใช้งานได้บนมือถือ
- มีข้อความขอความยินยอมในการใช้ข้อมูลส่วนบุคคลก่อนส่งข้อมูล

### 4. ลูกค้ากรอกฟอร์มขอราคา

ข้อมูลขั้นต่ำที่ควรเก็บ:

- ชื่อผู้ติดต่อ
- เบอร์โทร
- ยี่ห้อรถ
- รุ่นรถ
- ปีรถ
- ประเภทประกันที่สนใจ
- งบประมาณ
- หมายเหตุเพิ่มเติม
- แหล่งที่มา เช่น LINE OA, Rich Menu, Broadcast
- LINE User ID ถ้าระบบดึงจาก LIFF ได้

ข้อมูลที่ควรเพิ่มในระยะต่อไป:

- จังหวัด
- วันหมดประกันเดิม
- บริษัทประกันเดิม
- เลขทะเบียน
- ประวัติเคลม
- ต้องการซ่อมห้างหรือซ่อมอู่
- ความยินยอม PDPA

สิ่งที่ต้องตรวจสอบ:

- ฟิลด์บังคับต้องกรอกครบก่อนส่ง
- เบอร์โทรต้องอยู่ในรูปแบบที่ติดต่อกลับได้
- ระบบต้องกันการกดส่งซ้ำ
- ถ้าส่งข้อมูลไม่สำเร็จ ต้องแจ้งลูกค้าให้เข้าใจ
- ถ้าส่งสำเร็จ ต้องแสดงข้อความขอบคุณหรือหน้าขอบคุณ

### 5. ระบบบันทึก Lead

ในสถานะปัจจุบัน `car-insurance.js` บันทึกข้อมูลลง `localStorage` ก่อน ส่วนขั้นถัดไปควรส่งข้อมูลเข้า Google Sheets เหมือนระบบ CRM หลัก

Flow ที่ควรเป็น:

```text
ลูกค้ากดส่งฟอร์ม
-> ตรวจข้อมูล
-> สร้าง leadId
-> บันทึกลง localStorage สำรอง
-> ส่งข้อมูลไป Google Apps Script
-> Google Apps Script append row ลง Sheet MotorLeads
-> แจ้งผลกลับหน้าเว็บ
```

สถานะ lead เริ่มต้น:

```text
New lead
```

ข้อมูลที่ควรบันทึกใน `MotorLeads`:

```text
createdAt, leadId, source, campaign, lineUserId, name, phone,
carBrand, carModel, carYear, interestedPlan, budget, status, notes
```

สิ่งที่ต้องตรวจสอบ:

- หลังส่งฟอร์ม มีข้อมูลใหม่เข้า Google Sheets
- เวลาบันทึกถูกต้อง
- ข้อมูลภาษาไทยไม่เพี้ยนใน Sheet
- มี source/campaign เพื่อวัดผลแคมเปญ
- ถ้า Google Sheets ล่ม ระบบยังเก็บสำรองในเครื่องหรือแจ้งข้อผิดพลาด

### 6. ระบบแจ้งเตือนเจ้าหน้าที่

เมื่อมี lead ใหม่ ควรมีการแจ้งเตือนทีมขาย

ตัวเลือกการแจ้งเตือน:

- แจ้งใน Google Sheets
- ส่งอีเมล
- ส่ง LINE Notify หรือ Messaging API ไปยังกลุ่มทีมขาย
- แสดงในหน้า CRM dashboard

ข้อความแจ้งเตือนควรมี:

- ชื่อลูกค้า
- เบอร์โทร
- รถที่สนใจ
- ประเภทประกัน
- งบประมาณ
- แหล่งที่มา
- ลิงก์เปิด lead ในหลังบ้าน

สิ่งที่ต้องตรวจสอบ:

- มีการแจ้งเตือนทุกครั้งที่มี lead ใหม่
- ข้อมูลในแจ้งเตือนตรงกับข้อมูลที่ลูกค้ากรอก
- ไม่มีข้อมูลส่วนตัวเกินจำเป็นในช่องทางที่ไม่ปลอดภัย

### 7. เจ้าหน้าที่เปิด Lead ใน CRM

เจ้าหน้าที่เข้าหลังบ้านหรือ CRM เพื่อดู lead ใหม่

สิ่งที่เจ้าหน้าที่ต้องทำ:

- ตรวจข้อมูลลูกค้าและข้อมูลรถ
- ตรวจว่าข้อมูลครบพอออกใบเสนอราคาหรือไม่
- กำหนดผู้รับผิดชอบ
- เปลี่ยนสถานะ lead
- กำหนดเวลาติดตาม

สถานะที่ใช้:

```text
New lead
Contacted
Need more car info
Quoting
Proposal sent
Negotiating
Payment pending
Policy issued
Lost
Renewal
```

สิ่งที่ต้องตรวจสอบ:

- Lead ใหม่ต้องเห็นใน CRM
- ค้นหาด้วยชื่อหรือเบอร์โทรได้
- กรองตามสถานะได้
- แก้ไขข้อมูล lead ได้
- บันทึก note และวันติดตามครั้งถัดไปได้

### 8. เจ้าหน้าที่ติดต่อลูกค้า

ช่องทางติดต่อ:

- โทรศัพท์
- LINE OA chat
- ส่งข้อความ template
- นัดหมายให้ส่งเอกสารเพิ่มเติม

เป้าหมายของขั้นตอนนี้:

- ยืนยันข้อมูลรถ
- ถามวันหมดประกันเดิม
- ถามประวัติเคลม
- ถามความต้องการซ่อมห้าง/ซ่อมอู่
- ถามงบประมาณ
- ขอเอกสารที่จำเป็น

ถ้าข้อมูลยังไม่ครบ:

```text
สถานะ = Need more car info
สร้าง task = ขอข้อมูลรถเพิ่มเติม
```

ถ้าข้อมูลครบ:

```text
สถานะ = Quoting
สร้าง task = จัดทำใบเสนอราคา
```

สิ่งที่ต้องตรวจสอบ:

- ทุกครั้งที่ติดต่อลูกค้า ต้องบันทึกประวัติการติดต่อ
- ต้องบันทึกผลการคุย เช่น สนใจ, ยังไม่พร้อม, ขอคิดก่อน, ติดต่อไม่ได้
- ต้องมี next follow-up date

### 9. เจ้าหน้าที่จัดทำใบเสนอราคา

เจ้าหน้าที่นำข้อมูลไปเปรียบเทียบแผนจากบริษัทประกัน

ข้อมูลในใบเสนอราคาควรมี:

- บริษัทประกัน
- ประเภทประกัน
- ทุนประกัน
- เบี้ยสุทธิ
- เบี้ยรวม
- ซ่อมห้าง/ซ่อมอู่
- ค่าเสียหายส่วนแรก
- ความคุ้มครองบุคคลภายนอก
- รถหาย/ไฟไหม้/น้ำท่วม
- พ.ร.บ. รวม/ไม่รวม
- เงื่อนไขผ่อนชำระ
- วันหมดอายุใบเสนอราคา

หลังทำใบเสนอราคา:

```text
สถานะ = Proposal sent
บันทึกวันที่ส่งใบเสนอราคา
สร้าง task = ติดตามผลใบเสนอราคา
```

สิ่งที่ต้องตรวจสอบ:

- ใบเสนอราคาผูกกับ lead ถูกคน
- มีมากกว่า 1 แผนให้เปรียบเทียบได้ถ้าต้องการ
- ราคามีหมายเหตุว่าเป็นราคาประเมิน/รอยืนยันจากบริษัทประกันถ้ายังไม่ final
- ส่งใบเสนอราคาทาง LINE ได้

### 10. ลูกค้าพิจารณาและเลือกแผน

ลูกค้าสามารถ:

- เลือกแผนที่ต้องการ
- ขอเปรียบเทียบเพิ่ม
- ต่อรองราคา/เงื่อนไข
- ขอเวลาเพิ่ม
- ปฏิเสธ

ถ้าลูกค้าขอเวลา:

```text
สถานะ = Negotiating
สร้าง task = ติดตามอีกครั้ง
```

ถ้าลูกค้าปฏิเสธ:

```text
สถานะ = Lost
บันทึกเหตุผล เช่น ราคาแพง, มีเจ้าอื่นแล้ว, ยังไม่ซื้อ
```

ถ้าลูกค้าเลือกแผน:

```text
สถานะ = Payment pending
```

สิ่งที่ต้องตรวจสอบ:

- เปลี่ยนสถานะได้ถูกต้อง
- บันทึกเหตุผล lost ได้
- บันทึกแผนที่ลูกค้าเลือกได้
- มี task ติดตามถ้ายังไม่จบ

### 11. รับเอกสารและชำระเงิน

เอกสารที่อาจต้องใช้:

- สำเนาทะเบียนรถ
- กรมธรรม์เดิม
- บัตรประชาชน/เอกสารผู้เอาประกัน ตามเงื่อนไขบริษัท
- รูปถ่ายรถ ถ้าบริษัทประกันต้องการ
- หลักฐานชำระเงิน

ขั้นตอน:

```text
ลูกค้าส่งเอกสาร
-> เจ้าหน้าที่ตรวจเอกสาร
-> ลูกค้าชำระเงิน
-> เจ้าหน้าที่บันทึกหลักฐาน
-> ส่งเรื่องให้บริษัทประกัน
```

สิ่งที่ต้องตรวจสอบ:

- ไม่เก็บเอกสารสำคัญไว้ในที่ public
- มีสถานะเอกสารครบ/ไม่ครบ
- มีสถานะชำระเงินแล้ว/รอตรวจสอบ
- มีหมายเลขอ้างอิงการชำระเงิน
- มีวันที่เริ่มคุ้มครองที่ถูกต้อง

### 12. ออกกรมธรรม์

เมื่อบริษัทประกันยืนยันและออกกรมธรรม์:

```text
สถานะ = Policy issued
```

ข้อมูลที่ต้องบันทึก:

- เลขกรมธรรม์
- บริษัทประกัน
- วันที่เริ่มคุ้มครอง
- วันที่สิ้นสุดความคุ้มครอง
- ไฟล์กรมธรรม์หรือลิงก์เอกสาร
- ผู้รับผิดชอบ
- เบี้ยประกันสุทธิ/รวม
- ค่าคอมมิชชันถ้าต้องใช้รายงาน

สิ่งที่ต้องตรวจสอบ:

- ข้อมูลกรมธรรม์ตรงกับใบเสนอราคาที่ลูกค้าเลือก
- ส่งกรมธรรม์ให้ลูกค้าทาง LINE หรือช่องทางที่ตกลง
- ปิด task งานขายเดิม
- สร้าง task ต่ออายุล่วงหน้าอัตโนมัติ

### 13. บริการหลังการขาย

หลังออกกรมธรรม์แล้ว ระบบควรรองรับ:

- ส่งข้อความขอบคุณ
- แจ้งช่องทางติดต่อเมื่อเกิดเหตุ
- บันทึกคำถาม/ปัญหาหลังการขาย
- บันทึกเคลมเบื้องต้น
- แจ้งเตือนต่ออายุ

Reminder ต่ออายุที่แนะนำ:

```text
ก่อนหมดอายุ 60 วัน
ก่อนหมดอายุ 30 วัน
ก่อนหมดอายุ 7 วัน
วันหมดอายุ
```

สิ่งที่ต้องตรวจสอบ:

- มีวันสิ้นสุดกรมธรรม์
- ระบบสร้างงานต่ออายุได้
- ลูกค้าเดิมถูกแยกจาก lead ใหม่ได้
- ประวัติการขายเดิมยังดูย้อนหลังได้

- หลัง login ระบบกลับมาที่ LIFF Gateway ได้
- LIFF Gateway redirect ไป `car-insurance.html` ได้
- หน้าเว็บประกันรถยนต์แสดงผลถูกต้องบนมือถือ
- ฟอร์มกรอกข้อมูลได้ครบ
- กดส่งฟอร์มแล้วมีข้อความสำเร็จ
- ข้อมูล lead ถูกบันทึก
- ข้อมูลเข้า Google Sheets หรือ CRM
- เจ้าหน้าที่เห็น lead ใหม่
- เจ้าหน้าที่เปลี่ยนสถานะ lead ได้
- เจ้าหน้าที่บันทึก note และวันติดตามได้
- สร้างใบเสนอราคาได้
- ส่งใบเสนอราคาให้ลูกค้าได้
- บันทึกผลการตัดสินใจของลูกค้าได้
- บันทึกการชำระเงินได้
- บันทึกกรมธรรม์ได้
- สร้างงานต่ออายุได้

## จุดที่ระบบปัจจุบันทำได้แล้ว

- มีหน้าเว็บประกันรถยนต์ `car-insurance.html`
- มี LIFF Gateway สำหรับ login และ redirect
- มีลิงก์ LIFF ไปหน้าประกันรถยนต์
- มี CRM หลักสำหรับจัดการ lead แบบ localStorage
- มีตัวอย่าง Google Apps Script สำหรับรับ lead ประกันชีวิต

## จุดที่ควรพัฒนาต่อเพื่อให้ครบ Workflow

- เชื่อม `car-insurance.js` เข้า Google Sheets
- เพิ่ม sheet `MotorLeads`
- เพิ่ม LINE profile / LINE user id ในข้อมูล lead
- เพิ่มหลังบ้านเฉพาะประกันรถยนต์
- เพิ่มระบบจัดการโปรโมชั่น
- เพิ่มระบบใบเสนอราคา
- เพิ่มระบบรับเอกสารและสถานะชำระเงิน
- เพิ่มระบบกรมธรรม์และแจ้งเตือนต่ออายุ

## สถานะจบกระบวนการ

ถือว่ากระบวนการขายจบสมบูรณ์เมื่อ:

```text
ลูกค้าเลือกแผน
-> ชำระเงินแล้ว
-> เอกสารครบ
-> บริษัทประกันออกกรมธรรม์แล้ว
-> ส่งกรมธรรม์ให้ลูกค้าแล้ว
-> บันทึกวันหมดอายุ
-> สร้างงานต่ออายุในอนาคตแล้ว
```

ถ้าลูกค้าไม่ซื้อ ให้จบที่:

```text
สถานะ = Lost
บันทึกเหตุผล
กำหนดว่าจะติดตามใหม่ในอนาคตหรือไม่
```
